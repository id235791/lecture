# Git 전략 및 협업 규칙

Git은 협업 환경에서 소스코드를 관리하기 위한 필수 도구이며  
효율적인 개발을 위해 브랜치 전략, 커밋 규칙, PR 전략, 코드리뷰 방식을 함께 정의하는 것이 중요하다.  

프로젝트 규모가 커질수록 코드 충돌을 줄이고 안정적인 배포를 위해  
체계적인 협업 구조가 필요하다.  


---

## Git 브랜치 전략

Git은 브랜치를 활용하여 개발, 테스트, 배포를 분리하여 관리한다.  
대표적인 전략으로 Git Flow, GitHub Flow, Trunk-Based 방식이 있다.  


---

## Git Flow 전략

브랜치를 활용하여 개발과 배포를 명확하게 분리하는 방식이다.  

main / dev / feature 구조  


### 브랜치 구성

#### main 브랜치
운영(Production) 환경을 위한 브랜치이다.  

- 항상 배포 가능한 상태 유지  
- 실제 서비스 코드 관리  
- 직접 수정 금지  

#### dev 브랜치
개발 통합 브랜치이다.  

- feature 브랜치들을 병합하는 공간  
- 테스트 및 검증 수행  
- 다음 배포 준비  

#### feature 브랜치
기능 개발을 위한 브랜치이다.  

- dev 브랜치에서 생성  
- 하나의 기능 단위로 작업  
- 작업 완료 후 dev로 merge  


### 브랜치 흐름

main  
  ↑  
dev  
  ↑  
feature/login  
feature/signup  


### 작업 흐름

1. dev 브랜치에서 feature 브랜치 생성  
2. 기능 개발 진행  
3. feature → dev 병합  
4. 테스트 완료 후 dev → main 병합  
5. 배포 진행  


---

## 다른 브랜치 전략

### GitHub Flow

main / feature 구조  

- main 기준으로 PR 후 바로 배포  
- 구조가 단순하고 빠름  
- 스타트업 및 소규모 프로젝트에 적합  

### Trunk-Based Development

main 브랜치 하나만 사용  

- 짧은 주기의 커밋  
- 빠른 배포 가능  
- CI/CD 필수  


---

## 브랜치 전략 비교

### Git Flow
- 안정성 높음  
- 구조 명확  
- 관리 복잡  

### GitHub Flow
- 단순함  
- 빠른 개발 가능  

### Trunk-Based
- 매우 빠른 배포  
- 높은 팀 숙련도 요구

## 커밋 컨벤션 (Commit Convention)

커밋 메시지를 일정한 규칙에 따라 작성하는 방식이다.  
협업 시 변경 사항을 빠르게 이해하고 히스토리를 관리하기 위해 사용된다.  


### 기본 구조

타입: 제목  

본문 (선택)  

푸터 (선택)  


### 커밋 타입

- feat     : 새로운 기능 추가  
- fix      : 버그 수정  
- docs     : 문서 수정  
- style    : 코드 스타일 변경  
- refactor : 코드 리팩토링  
- test     : 테스트 코드  
- chore    : 기타 작업  


### 커밋 메시지 예시

feat: 로그인 기능 추가  
fix: 회원가입 오류 수정  
refactor: 사용자 서비스 구조 개선  


### 작성 규칙

- 제목은 50자 이내 작성  
- 명령형 문장 사용  
- 불필요한 설명 제거

## Pull Request (PR) 전략

PR은 개발자가 작성한 코드를 병합하기 전에  
검토를 요청하는 과정이다.  

코드 품질을 유지하고 협업을 원활하게 하기 위한 핵심 단계이다.  


### PR 기본 흐름

1. feature 브랜치 생성  
2. 기능 개발 및 커밋  
3. PR 생성 (feature → dev 또는 main)  
4. 코드 리뷰 진행  
5. 승인 후 merge  


### PR 작성 규칙

- 작업 내용을 명확히 작성  
- 변경 사항 요약 포함  
- 테스트 방법 작성  
- 관련 이슈 연결  


### PR 예시

[feat] 로그인 기능 추가  

- JWT 인증 로직 구현  
- 로그인 실패 처리 추가

## 코드리뷰 방식

코드리뷰는 PR을 통해 코드의 품질을 검증하는 과정이다.  

단순한 오류 확인뿐 아니라  
구조, 성능, 가독성까지 함께 검토한다.  


### 코드리뷰 기준

- 코드 가독성  
- 네이밍 규칙  
- 중복 코드 여부  
- 성능 문제  
- 보안 문제  


### 코드리뷰 규칙

- 비난이 아닌 개선 중심 피드백  
- 구체적인 수정 방향 제시  
- 작은 단위 PR 권장  
- 리뷰 후 반드시 반영  


### 코드리뷰 예시

잘못된 예  
이 코드 이상함  

올바른 예  
이 부분은 함수 분리하면 가독성이 좋아질 것 같습니다.  


---

## 전체 협업 흐름

feature 브랜치 생성  
→ 기능 개발  
→ 커밋 작성 (컨벤션 적용)  
→ PR 생성  
→ 코드리뷰 진행  
→ 수정 반영  
→ merge  
→ 배포  


---

## 주의사항

- main 브랜치 직접 수정 금지  
- 의미 없는 커밋 금지  
- PR 없이 merge 금지  
- 너무 큰 PR 지양  
- 브랜치 전략 팀 내 통일  


---

## 결론

Git 전략, 커밋 컨벤션, PR 전략, 코드리뷰 방식은  

- 협업 효율 향상  
- 코드 품질 유지  
- 안정적인 배포  

를 위한 핵심 요소이다.  

체계적인 협업 규칙을 통해  
프로젝트의 안정성과 유지보수성을 확보할 수 있다.
